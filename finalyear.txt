Chapter Three: Methodology/System Analysis and Design

3.1 Introduction to system analysis
System analysis is the structured process of understanding a problem domain, identifying weaknesses in the current process, eliciting requirements from stakeholders, and producing an implementable solution model. In this project, the problem domain is digital teaching and learning in a Faculty of Computing environment where internet connectivity may be unstable, expensive, or unavailable. The objective of system analysis was to design an offline-capable Learning Management System (LMS) that supports the complete teaching and learning cycle: course creation, enrollment, lesson delivery, assignments, quizzes, progress monitoring, discussions, certificates, and administration.

The analysis phase focused on three categories of stakeholders:
1. Students: need easy course discovery, enrollment, content access, assessment participation, and progress visibility.
2. Instructors/Teachers: need tools for course authoring, learner management, assessment setup, grading, announcements, and analytics.
3. Administrators: need institution-wide visibility, user governance, and platform health metrics.

The system analysis produced a role-based architecture implemented in Flask with SQLAlchemy and SQLite, with a practical focus on deployment simplicity and low hardware requirements. The final design also considered extensibility toward a more comprehensive Frappe LMS-compatible schema included in the repository (models_frappe_complete.py, routes_frappe_complete.py, main_frappe.py) for future scaling and migration.

3.2 Research methodology (Where applicable)
A design-and-build methodology (applied software engineering methodology) was used, combining elements of:
- Requirement elicitation through stakeholder-oriented needs analysis.
- Rapid prototyping to refine UI and workflow decisions.
- Iterative development and testing cycles.
- Comparative architecture review (existing Flask core versus Frappe-style complete model set).

Methodological steps used:
1. Problem identification:
   - Existing teaching workflows relied on fragmented manual tools.
   - Need for centralized and offline-capable LMS identified.
2. Requirement gathering:
   - Functional requirements (registration, course management, quizzes, assignments, dashboards, etc.).
   - Non-functional requirements (offline availability, responsiveness, role-based security, maintainability).
3. System modeling:
   - Data model design using entity relationships in SQLAlchemy.
   - Route and permission design for student/teacher/admin workflows.
4. Implementation:
   - Primary implementation in main_enhanced.py and models.py.
   - Initialization scripts for realistic sample data (init_db.py).
5. Validation/testing:
   - Functional route tests and manual role-based scenario testing.
   - Error-handler and edge-case checks (access denial, duplicate enrollment, invalid credentials).
6. Comparative enhancement planning:
   - Frappe-style model/API extension prepared for long-term migration.

Research approach justification:
- The project is an applied system development study, so iterative prototyping is suitable.
- Directly testing workflows with role-specific accounts provides practical validation.
- Building both baseline and advanced architecture options in one repository supports future upgrade feasibility analysis.

3.3 Description of the existing system
Before the proposed system, common instructional delivery in many local contexts can be described as semi-manual and fragmented. Typical characteristics include:
- Course materials shared via ad-hoc channels (email, messaging platforms, flash drives).
- Assignments submitted via unstructured media with limited traceability.
- Quiz and exam records stored separately from learning content.
- No reliable centralized progress tracking.
- Limited analytics for instructors and administrators.
- Weak continuity in low-connectivity/offline environments.

Challenges in the existing approach:
1. Data fragmentation:
   - Student records, grades, and learning content are dispersed across files and media.
2. Weak accountability:
   - Hard to confirm completion status, assignment timestamps, and grading fairness.
3. Low scalability:
   - As classes grow, manual handling becomes inefficient and error-prone.
4. Poor learner feedback loop:
   - Students do not receive immediate performance analytics in many manual systems.
5. Inconsistent access control:
   - No strict role-based permission model in informal platforms.

Within the repository context, an earlier simpler Flask structure existed but lacked full parity with enterprise LMS design principles. The redesigned project addressed this by introducing rich models, expanded routes, improved dashboards, and migration-ready architecture.

3.4 The Proposed System
The proposed system is an offline-first Faculty of Computing Learning System (FCLS), implemented primarily with Flask, SQLAlchemy, and SQLite. It is designed to provide end-to-end academic workflow support across student, teacher, and admin roles.

Core proposed capabilities:
1. User and role management:
   - Role types: student, teacher, admin.
   - Registration/login, profile management, role-based redirection and route guards.
2. Course lifecycle management:
   - Create, edit, publish, archive courses.
   - Structured metadata (category, difficulty, duration, summary, requirements).
3. Learning content delivery:
   - Lessons and optional chapter grouping.
   - Lesson navigation (previous/next) and completion tracking.
   - Resource and media upload support.
4. Assessment subsystem:
   - Quizzes with multiple question types and configurable pass threshold.
   - Attempt records, answer tracking, score computation, pass/fail status.
   - Assignments with student submissions, file uploads, grading, and feedback.
5. Enrollment and progress:
   - Course enrollment with duplicate prevention.
   - Lesson-based progress aggregation per course.
6. Extended engagement features:
   - Course discussions and replies.
   - Course reviews and rating.
   - Batch scheduling and live class management.
7. Certification:
   - Certificate issuance for qualifying learners (progress threshold logic).
8. Administration and monitoring:
   - User/course management and dashboard statistics.
   - Growth and participation indicators.

Implementation variants in repository:
- Operational enhanced app: main_enhanced.py + models.py + templates/.
- Advanced migration track: main_frappe.py + models_frappe_complete.py + routes_frappe_complete.py + frappe_lms templates.

This dual-track design supports immediate deployment while preserving an upgrade path to broader LMS feature parity.

3.5 Significance and Challenges of the proposed system
Significance:
1. Offline suitability:
   - Functions without internet dependency, ideal for constrained environments.
2. Educational process integration:
   - Unifies course delivery, assessment, feedback, and analytics in one platform.
3. Institutional value:
   - Enables data-driven monitoring for faculty-level decision-making.
4. Learner empowerment:
   - Students can self-monitor progress and performance.
5. Instructor productivity:
   - Structured tools for content authoring, grading, and engagement.
6. Scalability pathway:
   - Frappe-compatible data architecture included for future growth.

Challenges:
1. Security maturity:
   - Current implementation uses plain password comparison in core routes; production requires hashing and stronger auth controls.
2. Manual testing bias:
   - Automated test suites are not yet present in the repository.
3. SQLite scaling constraints:
   - Suitable for small-to-medium institutional use, but high concurrency requires migration to PostgreSQL/MySQL.
4. Data migration complexity:
   - Transitioning from baseline schema to complete Frappe schema requires controlled migration and validation.
5. Feature consistency across variants:
   - Maintaining parity between main_enhanced.py and main_frappe.py requires governance.
6. Operational hardening:
   - Additional measures needed for production (CSRF, rate limiting, secure secrets management, HTTPS).

3.8 System Design (Structural Design, Database Design and Specification, Interface Design)

3.8.1 Structural Design
The system follows a layered MVC-like Flask structure:
- Presentation Layer:
  - Jinja2 templates under templates/ and templates/frappe_lms/.
- Application Layer:
  - Route controllers in main_enhanced.py and routes_frappe_complete.py.
- Domain/Data Layer:
  - SQLAlchemy models in models.py and models_frappe_complete.py.
- Persistence Layer:
  - SQLite database (elearning.db / frappe_lms.db).
- File Storage:
  - uploads/ directory for profile images, lesson videos, assignment files.

Major structural modules:
1. Authentication and session control.
2. Role-based dashboard routing.
3. Course and lesson management.
4. Quiz lifecycle management.
5. Assignment lifecycle and grading.
6. Batch/live class coordination.
7. Discussions/reviews/certifications.
8. Admin monitoring and governance.

3.8.2 Database Design and Specification
Database technology:
- SQLite through Flask-SQLAlchemy ORM.

Primary entities in operational model (models.py):
1. User:
   - Identity, role, status, profile details.
2. Course:
   - Metadata, publication state, instructor relation.
3. Enrollment:
   - Student-course relationship with completion/progress fields.
4. Lesson / Lesson_Resource / Lesson_Progress:
   - Content unit, materials, and learner completion tracking.
5. Quiz / Quiz_Question / Quiz_Option / Quiz_Attempt / Quiz_Answer:
   - Assessment model with answer/score persistence.
6. Assignment / Assignment_Submission:
   - Instructor tasks and learner submissions with grading fields.
7. Announcement:
   - Course communication notices.
8. Chapter:
   - Logical course content grouping.
9. Batch / BatchEnrollment:
   - Cohort-based course delivery.
10. LiveClass:
   - Scheduled virtual class sessions.
11. Certificate:
   - Completion artifacts and verification identifiers.
12. Discussion / DiscussionReply:
   - Forum-like learner-instructor communication.
13. CourseReview:
   - Rating and textual feedback.

Database constraints and integrity features:
- Unique email per user.
- Unique student-course enrollment pair.
- Unique batch-student pair.
- Unique course-student review pair.
- Foreign key relationships and cascade deletion for dependent entities.

Design principles:

5. Teacher triggers certificate issuance.
6. System generates certificate record with certificate number and hash.
7. Student views certificate page.

Chapter Five: Conclusion & Recommendations

5.1 Conclusion
This project achieved its primary objective of delivering a comprehensive, offline-capable Learning Management System for a Faculty of Computing environment. The developed application integrates student learning, instructor content management, assessment workflows, and administrative governance in a single unified system.

Major accomplishments include:
1. Role-based access control and dashboard segmentation.
2. End-to-end course, lesson, quiz, and assignment functionality.
3. Progress tracking, discussion, review, batch, and certificate features.
4. Clear deployment pathway for local/offline environments.
5. Structured expansion route via Frappe-compatible architecture.

Therefore, the system is suitable as a practical institutional LMS prototype and can support real operational use in controlled environments.

5.2 Recommendation
To maximize impact and sustainability, the following recommendations are made:
1. Security hardening:
   - Implement password hashing (e.g., werkzeug.security), CSRF protection, stronger secret management, and optional MFA.
2. Automated testing:
   - Add unit, integration, and regression test suites to improve reliability during feature growth.
3. Database scaling:
   - Migrate from SQLite to PostgreSQL/MySQL for high concurrent multi-user deployment.
4. Observability:
   - Add structured logging, audit trails, and health monitoring dashboards.
5. Content tooling:
   - Improve rich content authoring and media processing workflows.
6. API standardization:
   - Expand REST API coverage and formalize API documentation.
7. DevOps readiness:
   - Add production-grade WSGI deployment configuration and backup automation.
8. Accessibility and UX:
   - Conduct formal usability studies and improve accessibility compliance.
9. Institutional integration:
   - Integrate student information systems and examination policy modules where needed.
10. Governance:
   - Establish release/version policy to manage dual architecture tracks.

5.3 Contribution to Knowledge

Chapter Three: Methodology/System Analysis and Design

3.1 Introduction to system analysis
System analysis is the structured process of studying an existing operational environment, identifying its limitations, extracting user and stakeholder needs, and translating those needs into technical and functional system requirements. In this project, system analysis was used to transform a basic educational web portal into a comprehensive Learning Management System (LMS) suitable for tertiary-level academic use, specifically for a Faculty of Computing context.

The analysis phase for this project focused on the following:
- Understanding academic workflow actors: administrators, instructors/teachers, and students.
- Mapping the complete learning lifecycle: course creation, enrollment, lesson consumption, assessment, assignment submission, grading, and certification.
- Defining constraints: offline-first operation, low infrastructure requirements, and local database persistence.
- Identifying implementation risks: insecure plain-text password handling, absence of advanced production hardening, and limited automated tests.

From the repository evidence, the system evolved through multiple versions:
1. A core Flask LMS implementation (main.py).
2. An improved and expanded implementation (main_enhanced.py + models.py + templates).
3. A parallel Frappe-LMS-inspired architecture (main_frappe.py, models_frappe_complete.py, routes_frappe_complete.py, init_frappe_lms.py).

Therefore, system analysis in this work was not a one-time static event, but an iterative engineering process that moved from simple CRUD functionality to a richer, role-based learning platform with broader educational features.

3.2 Research methodology(Where applicable)
The research methodology adopted is a hybrid of Design Science Research (DSR), Rapid Prototyping, and Iterative Incremental Development.

A. Design Science Orientation
The project is artifact-centric. The “artifact” is the offline LMS software, and the core research question is practical:
“How can a full-featured, low-cost, maintainable LMS be implemented for local academic environments without requiring cloud dependency or Docker orchestration?”

B. Iterative Development Process
The implementation reflects iterative cycles:
- Cycle 1: Build a baseline LMS with core login/course features.
- Cycle 2: Expand model schema and route logic for quizzes, assignments, and progress tracking.
- Cycle 3: Integrate advanced academic features such as chapters, batches, live classes, discussions, reviews, and certificate issuance.
- Cycle 4: Introduce a Frappe-LMS-compatible architecture to evaluate migration strategy and extensibility.

C. Prototype-and-Refine Workflow
Each cycle includes:
1. Requirement extraction from expected LMS capabilities.
2. Schema design in SQLAlchemy models.
3. Flask route implementation and role-based access restrictions.
4. Template-level user interface updates.
5. Seed data initialization (init_db.py / init_frappe_lms.py).
6. Manual integration testing through realistic role workflows.

D. Evaluation Approach
Evaluation was mainly empirical and scenario-based:
- Functional validation with seeded users (admin/teacher/student credentials).
- Workflow testing across enrollment, quiz, assignment, and dashboard analytics.
- Data persistence validation via SQLite-backed operations.
- Usability assessment through multi-template dashboard and module pages.

3.3 Description of the existing system
Before full enhancement, the existing system can be characterized as a minimal educational portal with limited structural depth. Its likely characteristics included:
- Basic user registration/login.
- Course listing and simple enrollment handling.
- Limited pedagogical tracking and weaker module decomposition.

Repository evidence confirms that the project later needed substantial expansion in:
- Database schema breadth (moving from basic entities to many interrelated educational entities).
- Role-specific operations.
- Learning progression analytics.
- Richer educational assets and assessment models.

Key shortcomings in the earlier baseline profile include:
1. Shallow educational structure: limited chapter/lesson hierarchy.
2. Limited pedagogy support: insufficient mechanisms for batch management, live classes, discussions, and granular assessment tracking.
3. Weak analytics depth: fewer institutional-level metrics and growth indicators.
4. Security constraints: non-hashed passwords and development-style configuration.
5. Scalability limits: mainly local/offline orientation with no production-grade deployment stack integrated.

Thus, the “existing system” served as a functional starter platform but required a full redesign/extension to satisfy realistic institutional LMS requirements.

3.4 The Proposed System
The proposed system is a comprehensive offline-first, role-based LMS implemented using Flask, SQLAlchemy ORM, Jinja templates, and SQLite persistence. It supports end-to-end academic learning operations for administrators, instructors, and students.

A. Core Design Goals

- Easy setup on standard laptops/desktops.
- Multi-role operations with access control boundaries.
- Comprehensive course lifecycle support.
- Extensible architecture for future migration/integration.

4. Assessment Engine

5. Academic Community Features

6. Credentialing

7. Administrative Monitoring

3.5 Significance and Challenges of the proposed system
Significance
1. Educational accessibility: The system can run in constrained environments where internet quality is poor or unstable.
2. Institutional autonomy: Departments can host and operate the LMS locally without cloud vendor lock-in.
3. Pedagogical completeness: Beyond courses, it supports quizzes, assignments, discussions, certificates, and live class scheduling.
4. Operational simplicity: Setup scripts (run.bat, run.sh) and SQLite reduce deployment complexity.
5. Research and prototype value: The architecture serves as a foundation for further AI, analytics, and adaptive learning extensions.

Challenges
1. Security hardening gap:
- Current logic uses direct password comparison in core routes.
- Production-grade hashing, CSRF defense, and strict security headers should be implemented.

2. Testing maturity:
- No dedicated automated test suite is present in repository structure.
- Quality assurance is predominantly manual/integration-based.

3. Production readiness:
- Flask development server invocation is used directly.
- Full production deployment pipeline (gunicorn/uWSGI + reverse proxy + TLS) is not embedded in project.

4. Scalability limitations:
- SQLite is excellent for small-to-medium local installations but less suitable for very high concurrency.

5. Data governance:
- Local/offline architecture demands strong backup discipline and storage management by operators.

Despite these challenges, the proposed system remains highly significant for educational institutions prioritizing low-cost deployability and practical function coverage.

4.2 Language Implementation
Primary implementation language is Python, with Flask used for web application routing, request handling, and template rendering. SQLAlchemy handles object-relational mapping and schema relationships.

Implementation mapping:
- Route/controller logic: main_enhanced.py (plus alternate entrypoints main.py and main_frappe.py).
- Data model layer: models.py (and models_frappe_complete.py for expanded parallel architecture).
- Initialization/seeding scripts: init_db.py and init_frappe_lms.py.
- UI layer: Jinja/HTML templates in templates directories.
- Automation scripts: run.bat and run.sh for setup and launch.

Data operations are implemented with SQLAlchemy sessions:
- Create operations through session add/commit.
- Read operations via query/filter/join/paginate patterns.
- Update/delete with transaction rollback handling on exceptions.

File handling is implemented with Werkzeug secure_filename and a configured uploads directory with extension checks and maximum size control.

4.3 Choice of Programming Language
Python was selected due to:
1. High developer productivity for rapid academic project delivery.
2. Strong ecosystem for web development, ORM, and educational prototypes.
3. Low barrier to maintenance and readability for student teams.
4. Broad support in local/offline setups.

Flask was chosen due to:
- Lightweight and flexible architecture.
- Easy route-based feature expansion.
- Seamless integration with Jinja and SQLAlchemy.
- Suitability for iterative and educational software prototyping.

4.4 Language justification
The language and framework choices are justified by the project’s constraints and goals:

A. Constraint alignment
- Requirement: easy deployment on local machines.
- Python + Flask + SQLite satisfies this with minimal external dependencies.

B. Educational maintainability
- Code readability is high, making handover and future student contribution easier.

C. Feature fit
- Flask handles session-based authentication, role branching, and module-level route decomposition effectively.

D. Development velocity
- Large functional surface (quizzes, assignments, certificate issuance, analytics) can be implemented quickly compared with more rigid frameworks.

E. Extensibility
- Presence of a parallel Frappe-inspired architecture demonstrates language/framework flexibility for migration-scale evolution.

B. Discussion
The system demonstrates that a robust institutional LMS can be developed with lightweight technologies under constrained infrastructure assumptions. This is particularly valuable in contexts where cloud dependency, licensing cost, or internet reliability is problematic.

The inclusion of both an enhanced implementation and a Frappe-inspired parallel architecture is a notable strength, showing forward-thinking design beyond minimum project completion.

Chapter Five: Conclusion & Recommendations

5.1 Conclusion
This project successfully designs and implements a comprehensive offline-first E-Learning Portal using Flask. The implemented system goes beyond a simple course listing platform and provides a complete educational workflow including authentication, role-based dashboards, course authoring, enrollment, lesson progression, assessment, assignment management, collaboration features, and certification.

From a systems perspective, the project achieves strong functional completeness for an academic LMS in a low-complexity deployment model. The use of Python, Flask, SQLAlchemy, and SQLite supports maintainability, readability, and practical institutional adoption.

Furthermore, the repository’s inclusion of migration-oriented architecture (Frappe-inspired models/routes) indicates strategic extensibility and architectural maturity. This increases the project’s long-term relevance beyond immediate demonstration.

In summary, the project meets its central objective: delivering a practical, feature-rich LMS suited for educational environments with limited infrastructure and a preference for offline/local execution.

5.2 Recommendation
To improve the system for institutional-scale or long-term deployment, the following recommendations are made:

1. Security hardening (highest priority)
- Replace plain-text password handling with secure hashing (e.g., werkzeug.security).
- Implement CSRF protection and stricter session policies.
- Externalize and rotate secret keys/environment configs.

2. Testing modernization
- Add automated tests (unit, integration, regression) using pytest.
- Create CI-based test execution and quality gates.

3. Deployment hardening
- Use production WSGI server and reverse proxy (gunicorn + nginx or equivalent).
- Enable HTTPS/TLS and structured logging.

4. Data and backup governance
- Introduce scheduled backup/restore scripts.
- Add audit logging for key administrative actions.

5. Scalability roadmap
- Migrate to PostgreSQL/MySQL for larger concurrency needs.
- Introduce caching for analytics-heavy pages if needed.

6. UX and accessibility improvements
- Standardize modern template system across all pages.
- Improve accessibility compliance (contrast, labels, keyboard navigation).

7. Analytics and reporting
- Add downloadable reports (student progress, course completion, grading trends).
- Include cohort/batch-level analytics and retention indicators.

8. Institutional integration options
- Optional integration with SSO/LDAP or school information systems.
- Add email/SMS notification channels where network permits.

4. Extends pedagogical system modeling in local context

5. Bridges educational software engineering and system design pedagogy
